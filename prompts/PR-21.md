## (1) Item A — Artifact deduplication

**The bug:** the same artifacts appear multiple times in the Artifacts grid (e.g. "Seed: Procestra Blog Idea" shown 3 times). This is an ingestion problem — artifacts are being added to the list more than once.

**Root cause candidates:**
- The artifact ingestion function is called multiple times on startup (e.g. `onAppear` + `FileWatcher` event + manual refresh = 3 calls)
- Each call appends without checking for existing entries with the same ID or path
- The `FileWatcher` fires once for each file on initial scan, and again on the first filesystem event

**The fix:**

### A1 — Idempotent ingestion

Before adding any artifact, check if it already exists:

```swift
func ingestArtifact(_ artifact: Artifact) {
    guard !artifacts.contains(where: { $0.id == artifact.id || $0.path == artifact.path }) else {
        return  // already present — skip
    }
    artifacts.append(artifact)
}
```

If artifacts don't have stable IDs based on path (only assigned at insert time), key uniqueness on `artifact.path` or `artifact.sourceURL` instead.

### A2 — Deduplicate on load

On project load, after ingesting all artifacts, deduplicate the array as a safety net:

```swift
var seen = Set<String>()
artifacts = artifacts.filter { artifact in
    let key = artifact.path ?? artifact.id.uuidString
    return seen.insert(key).inserted
}
```

### A3 — FileWatcher guard

If the `FileWatcher` fires an event for the same file multiple times (initial scan + first change event), guard the ingestion path to only run once per file path per session:

```swift
private var ingestedPaths = Set<String>()

func handleFileEvent(at path: String) {
    guard ingestedPaths.insert(path).inserted else { return }
    // ingest the artifact
}
```

```
- [ ] A1. Quit and relaunch. No duplicates appear.
- [ ] A1. FileWatcher events do not create additional duplicates.
```

## (2) Item B — Brand color correction

**The problem:** `prov/accent` (#1D9E75, green) is being applied to too many UI elements:
- Active tab underline/text in the project tab bar (Overview, Sources, Artifacts, Versions, Check-ins, Ledger, Export)
- The "Working" / "Stuck" / "Breakthrough" / "Paused" / "Done" status chip selected state (solid green fill)
- The "Save" action button label
- The "+ Add Artifact" button

When accent is everywhere, nothing has accent. Reserve it for primary actions.

**Read the current brand spec** (`procestra_ui_color_assignment.html`) before making any changes. Apply exactly what it specifies.

### B1 — Tab bar selected indicator

The selected tab should use `Brand.textPrimary` (near-black) for its text — NOT `prov/accent`. The active underline, if present, can use `prov/accent` as a 2pt bottom border, but the text itself must be `Brand.textPrimary` (selected) or `Brand.textSecondary` (unselected).

```swift
// Tab text:
.foregroundColor(isSelected ? Color(Brand.textPrimary) : Color(Brand.textSecondary))
// Active underline (optional):
Rectangle()
    .fill(isSelected ? Color("prov/accent") : Color.clear)
    .frame(height: 2)
```

### B2 — Status chips (Working / Stuck / etc.)

Status chips in the check-in row should use:
- **Selected:** `prov/tint-surface` background + `prov/accent` text — NOT solid green fill
- **Unselected:** `Color.clear` background + `Brand.textSecondary` text

```swift
Text(status.label)
    .foregroundColor(isSelected ? Color("prov/accent") : Color(Brand.textSecondary))
    .padding(.horizontal, 12).padding(.vertical, 6)
    .background(isSelected ? Color("prov/tint-surface") : Color.clear)
    .clipShape(RoundedRectangle(cornerRadius: 8))
```

### B3 — Primary CTA buttons

`prov/accent` solid fill IS correct for primary CTA buttons:
- "Save" (check-in save action) — solid `prov/accent` fill, cream text. **This is correct. Keep it.**
- "+ Connect Project" — solid `prov/accent` fill. **Keep.**
- "+ Add Artifact" — solid `prov/accent` fill. **Keep.**

The issue was not the buttons — it was status chips and tab text using the same solid accent treatment as buttons.

### B4 — Sidebar project status dots

The green dot next to project names in the sidebar (active project indicator) — keep these as `prov/accent`. They are a subtle accent moment, not overwhelming.

```
- [ ] B1. Tab bar selected tab: Brand.textPrimary text, optional 2pt prov/accent underline. Not green text.
- [ ] B2. Status chips: tint-surface bg + prov/accent text when selected. Clear bg + textSecondary when not.
- [ ] B3. Primary CTA buttons still use solid prov/accent fill. Not changed.
- [ ] B4. Sidebar status dots remain prov/accent.
```

## (3) Item C — White header regression

**The bug:** Provenance home screen intermittently shows a white/light-grey toolbar or header bar at the top of the content area. This was fixed in prior prompts but is regressing.

**Root cause:** the `toolbarBackground` modifier or the window's `backgroundColor` is not being applied in all code paths. When the view reloads, re-navigates, or the window is first shown, there's a brief window (or persistent state) where the macOS default white toolbar color shows through.

**The fix:**

### C1 — Force toolbar background at the window level

In the `NSWindow` setup:
```swift
window.backgroundColor = NSColor(named: "surface/base")
window.titlebarSeparatorStyle = .none
```

### C2 — Force toolbar background at the SwiftUI view level

On the root view of Provenance (not just the home screen — the outermost view):
```swift
.toolbarBackground(Color("surface/base"), for: .windowToolbar)
.toolbarBackground(.visible, for: .windowToolbar)
```

If these are only applied to the home tab and not to project views or other navigation states, the white shows through when switching. Apply to EVERY top-level view or to the root `ContentView`.

### C3 — NavigationStack / NavigationSplitView background

If Provenance uses `NavigationSplitView`, set:
```swift
NavigationSplitView {
    // sidebar
} detail: {
    // content
}
.background(Color("surface/base"))
```

And remove any `.background(.white)` or default `Color(.windowBackgroundColor)` that may be set on any view.

```
- [ ] C1. Open Provenance. Switch between Home, project tabs, and back to Home multiple times. No white header appears.
- [ ] C1. The header remains surface/base in all navigation states.
```

## (4) Item D — Home check-in textarea cursor alignment

**The bug:** in the Provenance home screen check-in textarea ("Ready to check in? Type a sentence or two."), the text cursor appears at a different horizontal position than the placeholder text.

**Apply the same SD-18 fix:**

1. On the NSTextView (or NSViewRepresentable wrapper):
   ```swift
   textView.textContainer?.lineFragmentPadding = 0
   textView.textContainerInset = NSSize(width: 8, height: 8)
   ```

2. On the placeholder text:
   ```swift
   Text("Ready to check in? Type a sentence or two.")
       .padding(.leading, 8)   // matches textContainerInset.width
       .padding(.top, 8)       // matches textContainerInset.height
   ```

3. Verify: cursor, placeholder leading edge, and typed text all align at x=8pt from the textarea's left inner edge.

```
- [ ] D1. Cursor and placeholder text are at the same horizontal position.
- [ ] D1. Typed text starts at the same x as the placeholder's leading character.
```

## (5) Item E — Seed trace files inside .ledger folder

**The current behavior:** `*.seed-trace.json` files are written to some location outside the `.ledger` folder (likely the project root or a sibling directory).

**The required behavior:** seed trace files must be stored INSIDE the project's `.ledger/` folder:
```
project/
  .ledger/
    *.seed-trace.json    ← here
    ledger.json
    snapshots/
```

**The fix:**

1. Find where the seed trace file path is constructed. Change the directory from `projectURL.appendingPathComponent("*.seed-trace.json")` (or wherever it currently lands) to `projectURL.appendingPathComponent(".ledger/\(filename).seed-trace.json")`.

2. If `.ledger/` doesn't exist when writing: create it.
   ```swift
   try FileManager.default.createDirectory(at: ledgerURL, withIntermediateDirectories: true)
   ```

3. Update the FileWatcher / seed-trace scanner to look for `*.seed-trace.json` inside `.ledger/` rather than the project root.

4. Migration: if any `*.seed-trace.json` files exist at the old location, move them to `.ledger/` on project load.

```
- [ ] E1. Transplanting a seed from Seed to a Provenance project writes the seed-trace file inside .ledger/.
- [ ] E1. FileWatcher picks up seed-trace files from .ledger/.
- [ ] E1. Old seed-trace files at the project root are migrated to .ledger/ on next project open.
```

## (6) Item F — Export: NSSavePanel instead of export folder

**The current behavior:** when the user exports from Provenance, it writes to a fixed "export" folder (likely `project/.ledger/export/` or `project/export/`). There is no prompt for where to save.

**The required behavior:**
- Show an `NSSavePanel` when the user initiates export
- The panel's default directory is the **project root folder** (not any subfolder)
- The panel's default filename is something reasonable: `[ProjectDisplayName]-provenance-export.[ext]`
- The user can navigate anywhere and change the filename
- No "export" subfolder is created automatically

**The fix:**

```swift
func exportProject(format: ExportFormat) {
    let savePanel = NSSavePanel()
    savePanel.allowedContentTypes = [format.contentType]
    savePanel.directoryURL = projectURL   // default to project root
    savePanel.nameFieldStringValue = "\(project.displayName)-provenance-export.\(format.extension)"
    savePanel.title = "Export \(project.displayName)"
    
    savePanel.begin { response in
        guard response == .OK, let url = savePanel.url else { return }
        try? exportProjectToURL(url, format: format)
    }
}
```

Remove any code that creates a fixed export directory or writes without user confirmation.

```
- [ ] F1. Clicking export opens an NSSavePanel.
- [ ] F1. Panel defaults to the project root folder.
- [ ] F1. Panel has a sensible default filename including the project name and format extension.
- [ ] F1. No automatic "export" subfolder is created.
```
