# PR-25 — Export reorganization + new PDF templates

The export tab gets its intent-first structure and two new PDF templates. Bundle for Works is dropped from the picker.

---

## (0) Read first (mandatory)

Before writing code, read in full:
1. `Procestra/Assets/Brand/provenance_export_layout_spec.md` — the authoritative layout spec for both PDFs. Reread until every section is clear (Shared structure, Authorship Report, Review Export, Implementation notes).
2. `Procestra/Assets/Brand/provenance_authorship_final.pdf` — visual reference for the Authorship Report. Open in Preview and study the spacing, the verified line, the timeline dot patterns, the Integrity section.
3. `Procestra/Assets/Brand/provenance_review_final.pdf` — visual reference for Review. Note: no teal, no integrity section, ochre status labels.

5. The current Export tab code in Provenance (the format picker view, the existing PDF export code, the Markdown export code, the Authorship Report code from PR-8).

**Critical rule:** the spec wins where it differs from the PDFs. The PDFs show intent; the spec gives measurements.

## (1) Item A — Reorganize the Export tab

### A1 — New picker structure

Replace the current 4-button format picker with an intent-first layout:

```
Format
  ○ Review                   ○ Authorship Report
    [ PDF ]  [ Markdown ]      [ PDF only ]
```

- The primary radio chooses the **intent** (Review or Authorship Report).
- A secondary control nested under the selected intent chooses the **format**:
  - Review: PDF or Markdown (segmented control or two buttons)
  - Authorship Report: PDF only — the format control is either absent or shows a single non-interactive "PDF" pill, with a tooltip "Authorship reports are PDF only."

### A2 — Description text under the picker

When Review is selected:
> "A printable or shareable summary of your project: description, check-ins, artifacts, and recent activity."

When Authorship Report is selected:
> "A ledger-backed PDF report proving authorship and process history. Used for forensic, legal, or archival purposes."

System UI 11pt, `text/secondary`.

### A3 — Will be included block

The existing "Will be included" block (check-ins, sources, artifacts counts) stays as-is. The "Include in export" toggles continue to drive what content lands in BOTH intents — same selection logic.

For Authorship Report, also show the existing "Ledger integrity: ✓ intact (chain since {date})" line (or "⚠ Chain broken at N" when it's broken).

### A4 — Promote to Works section

The "Promote to Works" section stays as a separate block below the Export button. Its design is unchanged. The internal bundle write path can stay as the implementation for Promote — but the user-facing format picker NO LONGER lists "Bundle for Works." See Item B.

```
- [ ] A1. Two-intent picker (Review / Authorship Report) replaces the four-button format picker.
- [ ] A2. Description text under picker matches selected intent.
- [ ] A3. "Will be included" block continues to drive content selection for both intents.
- [ ] A3. Ledger integrity status shown only when Authorship Report is selected.
- [ ] A4. Promote to Works section unchanged.
```

## (2) Item B — Remove Bundle for Works from the format picker

### B1 — Drop the option

Remove "Bundle for Works" from the user-facing format picker entirely. Most users don't know what to do with a `.provenance.bundle/` directory outside the Works workflow — exposing it as a generic format adds confusion.

### B2 — Keep the bundle code as the Promote implementation

The bundle-writing logic stays. It's now ONLY invoked by the "Promote to Works…" action (which writes the bundle to a known location and opens Works with it). No user-facing "Save as Bundle…" affordance for now — if archival users need it later, add a secondary button inside the Promote section.

### B3 — Works-side auto-detect (separate prompt, not this one)

The user's intent: when a user connects a folder in Works that contains a `.ledger/` directory, Works should auto-detect it and import metadata. This is a Works change — drafted separately as WK-N (not part of PR-25). Provenance's job here is just to make the Bundle for Works output less prominent.

```
- [ ] B1. "Bundle for Works" no longer appears in the Export tab format picker.
- [ ] B2. The bundle-writing logic still exists, invoked only by Promote to Works.
- [ ] B3. (Out of scope, but noted: WK-N follow-up handles auto-detect of .ledger in connected works.)
```

## (3) Item C — Implement the two PDF templates

Implement both PDF exports exactly per `provenance_export_layout_spec.md`. The spec is the authoritative source; the PDFs in `Assets/Brand/` show visual intent.

### C1 — Shared infrastructure

Build (or refactor existing) a PDF rendering layer in Swift / CoreGraphics that supports the shared structure both documents use:

- US Letter (612 × 792pt) pages
- Margins: 63pt L/R
- Top content starts at ~58pt from top
- Footer baseline at ~37pt from bottom
- **CoreGraphics coordinate inversion:** the spec measures from top (ReportLab convention); CG uses bottom-left origin. Wrap the page rendering in a top-down coordinate transform or compute Y values as `pageHeight - specY` everywhere. Be consistent — pick one approach.

### C2 — Typography

```swift
// Title:
NSFont(name: "NewYork", size: 28)?.bold ?? .systemFont(ofSize: 28, weight: .bold).withDesign(.serif)

// Author / metadata:
NSFont(name: "NewYork", size: 11) ?? .systemFont(ofSize: 11).withDesign(.serif)

// Body / check-ins:
NSFont(name: "NewYork", size: 10) ?? .systemFont(ofSize: 10).withDesign(.serif)

// Section eyebrows:
NSFont.systemFont(ofSize: 7)  // all-caps, kerned slightly

// Metadata labels:
NSFont.systemFont(ofSize: 7.5)  // or 8

// Stat numbers (Authorship summary):
NSFont(name: "NewYork", size: 20)?.bold

// Hash values:
NSFont(name: "Menlo", size: 8)  // monospace

// Footer:
NSFont.systemFont(ofSize: 6.5)
```

Use the NewYork font with serif system font fallback. Never substitute Times outside the prototype.

### C3 — Colors — token-driven, NO hex literals

Every color comes from `BrandTokens.swift`. Never write `#2E2D2A` or `NSColor(red:..., green:..., blue:...)` directly in the PDF code.

| Use | Token |
|---|---|
| Body text | `Brand.textPrimary` |
| Secondary text (author, dates) | `Brand.textSecondary` |
| Muted (eyebrows, metadata) | `Brand.textMuted` |
| Section rules (0.5pt) | `Brand.borderUI` |
| Top rule (1.2pt) | `Brand.textPrimary` |
| Teal — Authorship only | `Brand.provAccent` |
| Teal dark — Authorship only | `Brand.provAccentDark` |
| Timeline key dots (Authorship) | `Brand.provAccent` |
| Timeline key dots (Review) | `Brand.tan400` (or whatever the tan-400 token is called) |
| Timeline minor dots | `Brand.borderUI` |

If a token doesn't exist yet (e.g., `tan400`, `provAccentDark`), add it to `BrandTokens.swift` with the hex from the spec — but reference it by name in the PDF code.

### C4 — Header block (page 1)

Both documents:
1. **Eyebrow row** at `topMargin` — document type left ("Authorship Report" or "Review Export"), "Provenance · Procestra" right. System UI 7.5pt, `textMuted`.
2. **Heavy rule** (1.2pt, `textPrimary`) at 8pt below the eyebrow baseline.
3. **Title** (New York bold 28pt) at 22pt below the rule.
4. **Author / Medium · Date range** line at 18pt below the title — author left in New York roman 11pt, medium·date right.
5. **Document-specific line 5** at 18pt below line 4:
   - **Authorship:** verified line — teal filled circle (r=3.5pt, baseline +4), "Ledger chain intact since {date}" in System UI bold 8.5pt `provAccentDark`, then " · N chained events" in System UI 8.5pt `textMuted`. Plain inline, no box.
   - **Review:** activity line — "N versions · N check-ins · N artifacts · Connected {date range}" in System UI 8.5pt `textMuted`. No badge.
6. **0.5pt rule** in `borderUI` at 18pt below line 5.
7. **First section** starts 14pt below that rule.

### C5 — Section anatomy (every section, both docs)

```
[14pt gap]
Section eyebrow (System UI 7pt all-caps textMuted)
[14pt gap]
Section content
[content-specific spacing]
0.5pt borderUI rule
[14pt gap to next section eyebrow]
```

The 14pt gap between label and content is non-negotiable. Do not shrink it.

### C6 — Authorship Report sections (in order)

1. **Summary** — 4 equal columns: stat numbers (New York bold 20pt, textPrimary) above labels (System UI 7.5pt textMuted). E.g. "12 / Check-ins" "47 / Artifacts" etc. Columns aligned to a 4-column grid across the content width.
2. **Check-ins** — for each check-in: date and right-aligned `[STATUS]` label in `provAccentDark`; body in New York roman 10pt textPrimary. Vertical rhythm consistent.
3. **Process timeline** — vertical line down the left side of the content area; event entries with a colored dot, event name, right-aligned date, detail line below. Dot colors:
   - Project connected / Check-in / Promoted → `provAccent` filled circle
   - Snapshot / Artifact added → `borderUI` filled circle
4. **Integrity** — "Genesis hash" label (System UI 8pt textMuted) above hash value (Menlo 8pt textPrimary), 11pt gap. 14pt gap. "Chain head" label above its hash. Verification note (System UI 7.5pt textMuted) below chain head hash. No box, just plain text.

### C7 — Review Export sections (in order)

1. **Description** — "Medium" label inline with value; Working Description paragraph (New York roman 10pt textPrimary); Intent paragraph below.
2. **Check-ins** — same layout as Authorship, but the right-aligned status label is in `Brand.ochre600` (the user mentioned "ochre-600" in the spec — add the token if absent), not teal.
3. **Artifacts** — for each: name (New York roman 10pt textPrimary), type and date right-aligned (System UI 7.5pt textMuted), description below in System UI 8.5pt textMuted.
4. **Process timeline** — same shape as Authorship, but ALL dots in `Brand.tan400`. No teal distinction in Review.

### C8 — What Review must NOT have

- No verified badge / teal chain elements
- No hash values
- No Integrity section

### C9 — Footer (both, all pages)

- Left: `"{Document type} · {Title} · {Author}"`
- Right: `"Generated by Provenance · {date}"`
- 0.4pt `borderUI` rule 11pt above baseline
- System UI 6.5pt `textMuted`

### C10 — Page overflow

When content approaches 1.4in (~100pt) from the bottom: break to a new page. New pages start with a 1.2pt `textPrimary` rule at 0.9in (~65pt) from the top, then content continues. No header block on continuation pages.

### C11 — Markdown export for Review

The Review intent can also export as Markdown. The same section structure applies:

- Header block → YAML frontmatter (title, author, medium, date_range, versions, check_ins, artifacts, connected)
- Section eyebrows → H2 (`## Description`, `## Check-ins`, `## Artifacts`, `## Process timeline`)
- Section content → markdown body
- 0.5pt rules → `---` horizontal rules
- Footer → not rendered in Markdown (the file has no pagination)
- Timeline dots → not rendered (replaced by a `- ` list item for each timeline event)

Authorship Report does NOT get a Markdown variant — the picker enforces PDF only for that intent.
