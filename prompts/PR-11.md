# PR-11 — Sidebar project deduplication

Bug: Provenance sometimes shows duplicate entries for the same project in the sidebar. The user suspects this is related to iCloud sync (the project's folder showing up under two paths, both connected separately at different times).

PR-10 introduced `projectId` (a stable UUID stored in `.ledger/manifest.json`) as the canonical project identifier. PR-11 uses it to prevent duplicate sidebar entries.

## (1) Load-time deduplication

In `Services/ProjectStore.swift`, after loading the projects list from persistent storage:

- Group entries by `projectId`.
- For each group with more than one entry:
  - Keep the entry whose folder resolves successfully (via the PR-10 ProjectLocator).
  - If multiple resolve, keep the one with the most recent `lastActivity` timestamp.
  - Discard the others — but do NOT delete their stored state from disk; just don't display them in the sidebar.
- Log a `projectsDeduplicated` ledger event (new LedgerEvent.type case) with detail: `Removed N duplicate project entries (kept projectId <uuid>)`. One event per dedup pass, not per duplicate.

The dedup happens at every project list load — startup, after Connect Project, after a project is renamed, etc. Cheap and idempotent.

## (2) Connect-time guard

When the user connects a new project folder via the Connect Project flow (PR-9), check whether a project with that `projectId` is already in the store:

- Read the candidate folder's `.ledger/manifest.json` to get its projectId.
- If the projectId already exists in the store:
  - Don't add a second entry.
  - Update the existing entry's folder path/bookmark to the new location (matches PR-10's section 9 duplicate-projectId rule).
  - Show an inline confirmation: "Updated location for '<Display Name>' — was already tracked under this project ID."
- If the projectId is new: add as today.

## (3) Sidebar UI — no change needed

After dedup, the sidebar simply renders the deduped list. No visual differences from today's sidebar when there are no duplicates. The user just stops seeing the doubled entries.

## (4) iCloud-specific note

If a project's folder is in iCloud Drive and the user has connected it from two different paths (e.g., once from `~/Library/Mobile Documents/...` and once from `~/iCloud Drive/...` which can resolve to the same folder), the dedup still works because both manifest reads return the same projectId. The user effectively gets the most-recent path as the canonical one, and the duplicate is silently dropped.

We do NOT add iCloud-specific logic. The projectId-based dedup handles all the cases generally.

## Acceptance

- A project store containing two entries with the same projectId (simulated by manually adding the same path twice, or by an iCloud-driven duplicate) loads with only one entry shown in the sidebar.
- The kept entry is the one whose folder resolves successfully (or, if multiple resolve, the one with the most recent activity).
- A `projectsDeduplicated` ledger event appears on dedup.
- Connecting a folder whose projectId is already tracked → no new entry; existing entry updates to the new path.
- Single-project case (no duplicates) → no behavior change.

## Don't

- Don't delete the discarded entries from persistent storage. Just hide from the sidebar.
- Don't try to detect or fix iCloud sync issues at the filesystem level — that's out of scope.
- Don't add UI for "merge duplicates" or similar power-user controls. Dedup is automatic.
- Don't change the ProjectStore's serialization format.
- Don't dedupe across different projectIds even if their names match — name collision is fine, projectId is identity.
