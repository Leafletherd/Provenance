## (1) Item A — Header regression OR gradient

The project view header is showing as a different color than the surrounding background again. Two acceptable fixes — pick ONE:

### A — Option 1: re-apply PR-24's pattern

Make the project view background continuous with the Home view's, as PR-24 specified. No header band, just the tab bar floating over the unified `Brand.surfaceBase` background, with the 0.5pt `Brand.borderUI` divider below the tabs.

### A — Option 2: gradient seam

If the header has to remain a distinct color (e.g., for vibrancy reasons or to mark the project view), make it gradient into the surrounding background from the sidebar edge inward. The user's words: "the header color gradiates into the background from around the sidebar so it looks seamless."

Implementation:
```swift
LinearGradient(
    colors: [Brand.surfaceSidebar, Brand.surfaceBase],
    startPoint: .leading,
    endPoint: UnitPoint(x: 0.25, y: 0.5)  // gradient resolves to surfaceBase within first quarter
)
```

The gradient starts at the sidebar's right edge (where the sidebar tan ends) and resolves to the main content background within the first ~25% of the header width. The right portion of the header is indistinguishable from the content background.

Pick whichever approach is simpler given the current architecture. Document the choice in the ship report.

```
- [ ] A1. Header reads as seamless against the content background. No abrupt color change.
- [ ] A1. Approach used (continuous bg OR gradient) documented in ship report.
```

## (2) Item B — Tab text vertical centering

**The bug:** the Overview / Sources / Artifacts / etc. tab labels are not equidistant from the top and bottom of the tab bar.

**The fix:** the tab bar container has equal top and bottom padding; the tab text is vertically centered within it.

```swift
HStack(spacing: 4) { /* tabs */ }
    .padding(.horizontal, 16)
    .padding(.vertical, 8)       // SAME top and bottom
    .frame(maxWidth: .infinity, alignment: .leading)
    // The tab buttons inside have NO additional vertical asymmetry:
```

Each tab button:
```swift
Text(tab.label)
    .padding(.horizontal, /* chip horizontal */)
    .padding(.vertical, /* chip vertical, equal top and bottom */)
```

Verify in the installed build: measure the gap above the tab text and below the tab text — they must be identical.

```
- [ ] B1. Tab text is vertically centered within the tab bar.
- [ ] B1. Top and bottom gaps measured to be identical.
```

## (4) Item D — Format picker visual consistency between intents

**The bug:** in Authorship Report, the "PDF only" output indicator looks different from the PDF/Markdown segmented toggle in Review. They should look like the same control, just with different state.

**The fix — option preferred:** show the segmented control identically in both intents. In Authorship, the "Markdown" segment is rendered DISABLED — same shape, same position, same colors as in Review, but with reduced opacity (50%) and `.disabled(true)` so it can't be tapped. PDF stays selected. A subtle tooltip on the Markdown segment in Authorship: "Authorship reports are PDF only."

```swift
HStack(spacing: 4) {
    SegmentButton(label: "PDF", isSelected: format == .pdf, isEnabled: true) { format = .pdf }
    SegmentButton(label: "Markdown",
                  isSelected: format == .markdown,
                  isEnabled: intent == .review) { format = .markdown }
        .opacity(intent == .review ? 1 : 0.5)
        .disabled(intent != .review)
        .help(intent == .review ? "" : "Authorship reports are PDF only.")
}
```

Visual outcome: the picker structure is identical regardless of intent — only the enabled/disabled state of Markdown changes. Users can see at a glance that PDF is the only option for Authorship without a different control shape.

```
- [ ] D1. Format picker uses the same segmented control in both intents.
- [ ] D1. Markdown segment is disabled (not absent) when Authorship is selected.
- [ ] D1. Tooltip explains why Markdown is disabled.
```

## (5) Item E — Button design philosophy

**Goal:** lock in a small, named set of button categories. Document them in `Brand/PROCESS_SUITE_DESIGN_STANDARDS.md` (or a new `BUTTON_SYSTEM.md` in `Assets/Brand/`). Every button instance in the app must use one of these — no ad-hoc styling.

### E1 — Four categories

| Category | Use | Style |
|---|---|---|
| **Primary CTA** | The single most important action on a screen (Save, Connect Project, Add Artifact, New Check-in, Export) | Solid `prov/accent` fill (or per-app accent), cream text (`Brand.surfaceBase`), 8pt corner radius, 12pt H × 8pt V padding |
| **Label-action** | Inline links / secondary affordances (Reveal in Finder, Edit Context, Refresh, Promote to Works…) | No fill, `Brand.textBrand` (tan) text, no border, 8pt corner radius on hover bg (`Brand.surfaceHover`), 8pt H × 4pt V padding |
| **Pill / chip / tab** | Selectable state indicators (filter chips, project tabs, segmented controls) | Selected: `Brand.surfaceSidebar` (or tint) fill + `Brand.textPrimary` text. Unselected: clear + `Brand.textSecondary` text. 8pt corner radius, 12pt H × 6pt V padding (equal top/bottom) |
| **Icon-only** | Hover-affordance icons (sidebar collapse, toolbar icons, X-on-card) | `Brand.textSecondary` icon at default, `Brand.textPrimary` on hover, circular `Brand.surfaceHover` bg on hover, 24×24pt frame |

### E2 — Apply across the app

Walk the app and confirm every interactive control fits one category. For any control that doesn't fit cleanly, force-fit it into the closest category — don't invent new ones.

Specifically verify and fix as needed:
- Sidebar `+` / `-` buttons (project list bottom) → Icon-only
- Sidebar Home row → Pill (selected state)
- Sidebar project rows → Pill (selected state)
- Tab bar items (Overview/Sources/etc.) → Pill (per PR-24, now formally categorized)
- Filter chips (All/Seeds/Notes/etc.) → Pill
- Export tab format toggle → Pill (segmented, two adjacent Pills)
- "Reveal in Finder" / "Edit Context" / "Refresh" → Label-action
- "+ Connect Project" / "Save" / "+ Add Artifact" / "+ New Check-in" / "Export" → Primary CTA
- "Promote to Works…" (becomes "Open in Works" per Item F) → Label-action
- "Include in export" checkboxes → Native macOS checkbox (separate, not a button)
- Status chips on check-ins (Working/Stuck/etc.) → Pill

### E3 — Document the system

Write the categories into `Procestra/Assets/Brand/BUTTON_SYSTEM.md` (new file). Future prompts and design sessions reference this file.

```
- [ ] E1. Four button categories defined and documented.
- [ ] E2. Every interactive control in Provenance fits one of the four.
- [ ] E3. `BUTTON_SYSTEM.md` written and committed.
```

## (6) Item F — Drop the Works bundle entirely

**Decision:** drop the `.provenance.bundle/` format and the "Promote to Works" workflow as a packaging step. Replace with "Open in Works" — a button that launches Works pointing at the same project folder. Works will detect `.ledger/` automatically (WK-N follow-up).

### F1 — Remove bundle code

Delete:
- `BundleExporter.swift` (or whatever writes `.provenance.bundle/`)
- The `.provenance.bundle/` format writer for `manifest.json`, `process.json`, etc.
- Any `BundleFormat` enum case in export logic
- Tests for bundle export

Keep `.ledger/` as-is — that's the canonical record.

### F2 — Replace "Promote to Works" with "Open in Works"

The section on the Export tab labeled "Promote to Works" becomes "Open in Works":

```swift
VStack(alignment: .leading, spacing: 8) {
    Text("Open in Works")
        .font(.system(size: 13, weight: .semibold))
    Text("Open this project folder in the Works app to add it to a portfolio. Works will read your provenance data automatically.")
        .font(.system(size: 11))
        .foregroundColor(Brand.textSecondary)
    Button(action: openInWorks) {
        Text("Open in Works…")
    }
    .buttonStyle(LabelActionButtonStyle())  // per E1
}
```

The action launches Works with the project folder URL via the `works://add?path=...` URL scheme (already implemented in WK-5). Works receives the folder, detects `.ledger/`, populates its UI.

### F3 — Schema and compatibility

The `.ledger/` format is unchanged. `manifest.json`, `seed-trace/*.json`, snapshots all stay. Only the bundle wrapper goes.

If there are any existing `.provenance.bundle/` directories in the wild (from older versions), they remain readable by Works (if Works still has the import code) but Provenance will no longer create new ones. WK-N will eventually remove bundle-reading from Works once auto-detect is stable.

### F4 — Documentation

Update `SUITE_FORMAT_CONTRACTS.md` — Contract B (`.provenance.bundle/`) is deprecated. Add a note that `.ledger/` IS the contract going forward.

```

- [ ] F3. .ledger/ format unchanged.
- [ ] F4. SUITE_FORMAT_CONTRACTS.md updated: Contract B deprecated.
```
