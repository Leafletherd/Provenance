## (1) Lock Home-page elements together during resize

**Current bug:** When the user resizes the Provenance window, the Home page's elements drift out of alignment with each other. Specifically: the greeting ("Good morning, …"), the prompt question, the text box (check-in input), and the project selector below all reposition somewhat independently — so a wider window splays them and a narrower window crowds them inconsistently.

**Required:** All four elements live in a **single vertical stack** that maintains its internal alignment as a unit. When the window resizes, the stack as a whole repositions (per its centering rules), but the relative offsets between elements stay locked.

**Implementation:**

- In `Sources/Views/Home/HomeView.swift`, wrap greeting + prompt + text box + project selector in a single `VStack` with a consistent `.frame(maxWidth: ...)`
- The VStack centers horizontally within the available width (`.frame(maxWidth: .infinity)` on its parent)
- Internal alignment of the VStack: see (2) — decide centered vs leading
- The VStack has fixed internal spacing (e.g., 16pt between greeting and prompt, 12pt between prompt and text box, 12pt between text box and project selector — or whatever spacings look right; fix once and lock)
- The VStack has a maxWidth (e.g., 560pt) — past that, the text box stops growing and the stack stays at the chosen width centered in the window

**Verify:** Resize the window from minimum (~800pt wide) to maximum (full screen). The four elements should move together as a unit, never overlap, never re-flow relative to each other.

## (2) Alignment decision — left-align all four elements

The user asked: "decide — center the prompt with the text box and good morning, OR left-align all three." They also said the project selector below should be "left-aligned with the prompt and the text box."

**Decision: LEFT-ALIGN all four elements (greeting, prompt, text box, project selector) within the VStack.**

Rationale: the text box is intrinsically a left-aligned form element (cursor starts on the left, content reads left-to-right). Center-aligning a multi-line text box looks awkward. Left-aligning everything to the same leading edge gives a clean column read.

**Implementation:**

- VStack alignment: `.leading`
- Every child element's `.frame(maxWidth: .infinity, alignment: .leading)` so text doesn't drift
- The VStack itself is horizontally **centered** in the window (via `.frame(maxWidth: .infinity)` on its parent + the VStack's own `.frame(maxWidth: 560)` so it's a content column centered on the page)

**End result:** A 560pt-wide content column, centered horizontally in the window. Inside that column, every line starts at the same leading edge.

## (3) Text box placeholder text positioning

**Current bug:** The placeholder/hint text inside the check-in text box is rendering inline with where the user's cursor lands when they start typing — but the user reports it's positioned wrong (likely the placeholder is shifted from where the actual typed text appears).

**Cause (probable):** The placeholder is overlaid on the text box with different padding/insets than the actual `TextField` / `TextEditor` content. Or the placeholder is positioned absolutely while the input has its own padding.

**Fix:**

- Find the check-in text box implementation in `HomeView.swift` (likely a `ZStack` overlay pattern with a placeholder `Text` over a `TextEditor`)
- Ensure the placeholder `Text` has the **exact same padding** (top, leading, trailing, bottom) as the `TextEditor`/`TextField`
- Use a consistent font and lineHeight between placeholder and input
- When the user starts typing, the first character should appear at the EXACT same x,y where the placeholder's first character was

**Verify:** Click into the text box. Note where the cursor lands. Type one character. That character should appear in the same position where the placeholder's first letter was.

If using a SwiftUI `TextField` with `prompt:`, the placeholder positioning is automatic and this should already be correct — in which case the bug is likely a custom overlay implementation that needs to be removed/replaced with the native `prompt:` parameter.

## (4) Project selector under prompt — left-align with prompt and text box

**Current state:** The project selector (the chip / button that picks which project the check-in is targeted at) sits below the text box but is offset from the prompt/text-box leading edge.

**Fix:** Per (2), with the whole VStack left-aligned, this falls out naturally — the project selector lives in the same VStack and is left-aligned to the same leading edge.

**Verify:** The leading edge of the greeting text, the prompt question text, the text box's left edge, and the project selector's left edge should all sit on the same vertical line.

## (5) Projects tab — click goes to project, no longer toggles prompt target

**Current behavior:** In the Projects tab (the tab listing all connected projects on the Home view, per PR-13's project-aware prompt), clicking a project row toggles whether that project is the check-in's prompt target.

**New behavior:** Clicking a project row should **navigate to that project** (same as opening the project from the sidebar — switch the app's selected project to the clicked one and switch to the Overview tab). The check-in prompt target is determined by which project is currently selected app-wide, not by a separate toggle in this list.

**Implementation:**

- Find the Projects tab view (likely `Sources/Views/Home/ProjectsTabView.swift` or inline in `HomeView.swift`)
- Find the per-row tap handler. It currently sets some `selectedProjectForCheckin` state or similar.
- Replace the tap handler with the same action used when a project is clicked in the sidebar (probably routes through a `ProjectStore.openProject(_:)` / `AppState.selectedProject = ...` + tab switch to Overview)
- Remove any UI affordance that indicated "this row is the current check-in target" (highlighted state, checkmark, etc.) — that's now redundant since selection = navigation

**Edge case:** If the user is on the Home view (no project open) and clicks a project, the app navigates into that project's view (Overview tab). The Home view becomes inaccessible until they navigate back via the Home button (which already exists in the toolbar).

## (6) Remove the "›" chevron from project tabs

**Current state:** Each project tab/row on the Home view's Projects tab has a "›" (chevron) glyph indicating "click to expand" or "click to navigate."

**Fix:** Remove the "›" entirely. With (5), the row is implicitly clickable; the chevron is redundant decoration.

**Implementation:**

- Find the per-row view in the Projects tab
- Remove the `Image(systemName: "chevron.right")` or whatever glyph is being rendered at the trailing edge
- Adjust the row's trailing padding so the right edge isn't awkwardly empty (probably reduce trailing padding by ~8pt)
