Refinements after PR-22 landed. Closer, but four specific issues remain.

## (1) Item A — Sidebar background extends under the header

**The bug:** the sidebar's tan background does NOT extend up under the title bar / window chrome region. There's a visible seam between the title bar area and where the sidebar background begins.

**The fix:** match Seed's sidebar pattern from SD-11/PR-16, which uses a ZStack with `Brand.surfaceSidebar.ignoresSafeArea()` painting the FULL column including under the title bar:

```swift
// Provenance sidebar root view:
ZStack(alignment: .top) {
    Brand.surfaceSidebar
        .ignoresSafeArea()  // Critical: paints under the title bar
    
    // Sidebar content
    VStack(spacing: 0) {
        Color.clear.frame(height: 28)  // top inset for under-title-bar
        // ... rest of sidebar
    }
}
```

If Provenance's sidebar uses `NavigationSplitView`, the sidebar column may need an additional modifier to extend under the title bar:

```swift
.navigationSplitViewColumnWidth(min: 220, ideal: 260)
.toolbar(removing: .sidebarToggle)  // if needed
```

Also check that no system `NSVisualEffectView` is overriding the tan background (per the SeedSidebarMaterialKiller pattern from SD-11):

```swift
private struct SidebarMaterialKiller: NSViewRepresentable {
    func makeNSView(context: Context) -> NSView { NSView() }
    func updateNSView(_ nsView: NSView, context: Context) {
        DispatchQueue.main.async {
            var v: NSView? = nsView
            while let current = v {
                if let effect = current as? NSVisualEffectView {
                    effect.material = .windowBackground
                    effect.state = .inactive
                    effect.blendingMode = .behindWindow
                }
                v = current.superview
            }
        }
    }
}
```

Plant `SidebarMaterialKiller().frame(width: 0, height: 0)` inside the sidebar's ZStack to defeat any NSVisualEffectView that might be tinting the title bar region differently from the sidebar body.

```
- [ ] A1. Sidebar tan background extends fully from the bottom of the sidebar to the top of the window, under the title bar.
- [ ] A1. No seam visible between the title bar area and the sidebar body.
- [ ] A1. Light and dark mode both show consistent sidebar extent.
```

## (2) Item B — Tile backgrounds match the sidebar tan (light mode)

**The bug:** after PR-22, tile backgrounds were changed to `Brand.surfaceBase` (cream). The user now wants them to match the **slightly lighter tan color of the sidebar** — same as `Brand.surfaceSidebar` (or whatever token paints the sidebar background).

### B1 — Identify the sidebar token

Read the sidebar background color. This is likely `Brand.surfaceSidebar` (#EFE8DA or similar — slightly lighter than the main canvas cream `surfaceBase`).

### B2 — Apply to tiles

In light mode, every content tile uses this sidebar tan as its background, with `Brand.borderSubtle` as the border:

- **Artifact tiles** (Artifacts tab cards)
- **Check-in tiles** (Check-ins tab entries)
- **Source tiles** (Sources tab)
- **Version tiles** (Versions tab)
- **Manuscript rows** (Overview tab manuscript list)
- **Project context box** (Overview tab — Working Description / Intent area)
- **Stat tiles** (the "13 versions / 2 check-ins / 0 sources / 14 artifacts" — these already use surfaceSunken which is OK; if they look too dark, swap to surfaceSidebar)

```swift
// For tile backgrounds:
.background(Brand.surfaceSidebar)  // NOT surfaceBase, NOT surfaceSunken
.overlay(
    RoundedRectangle(cornerRadius: 8)
        .stroke(Brand.borderSubtle, lineWidth: 0.5)
)
```

The main content area background (around the tiles) stays `Brand.surfaceBase` (cream) — only the TILE rectangles get the sidebar tan.

```
- [ ] B1. Light-mode tile backgrounds = Brand.surfaceSidebar (sidebar tan).
- [ ] B2. Main content area around tiles = Brand.surfaceBase (cream).
- [ ] B2. All tabs walked (Overview, Sources, Artifacts, Versions, Check-ins) — every tile uses the new tan.
```

## (4) Item D — Sidebar selection: press state matches settled state

**The bug:** when clicking a sidebar item, there's a brief moment during the press where the highlight looks different from how it looks after release (settled selected state). The press state should match the settled state — same color, same shape, no flicker.

### D1 — Identify the press state

SwiftUI `List` and NSTableView both have separate visual treatments for:
- **Press / pressed** — while the mouse button is held down
- **Selected** — after release, while the row remains selected

These can use different colors (e.g. system uses `controlAccentColor` during press and `unemphasizedSelectedContentBackgroundColor` for settled).

### D2 — Force them to match

Make the press state visually identical to the settled state:

```swift
// In SidebarRow:
.background(
    isSelected || isPressed
        ? RoundedRectangle(cornerRadius: 6).fill(/* settled selection color */)
        : nil
)
```

Track press via `.onLongPressGesture(minimumDuration: 0)` or `gesture(DragGesture(minimumDistance: 0).onChanged { ... }.onEnded { ... })`, OR use a custom `ButtonStyle` that sets `isPressed` to drive the background.

If using SwiftUI's built-in `List(selection:)`, override `.listRowBackground` to use only the settled-state color and not the system pressed-state color:

```swift
.listRowBackground(
    isSelected ? Color("...") : Color.clear
)
```

If NSTableView is in use under the hood, set:
```swift
tableView.selectionHighlightStyle = .none
// And paint the highlight manually via the row view, so press and selected paint identically
```

### D3 — Verify

Click a project row in the sidebar. While holding the mouse button down, the highlight must look identical to how it looks after releasing the button. No flicker, no color shift, no shape change.

```
- [ ] D1. Identified what was different between press and settled states.
- [ ] D2. Press state forced to match settled state.
- [ ] D3. Clicking a sidebar row: no visible flicker between press and release.
```
