# PR-27 — Project view = Seed bench chrome pattern + dual-copy export

The Home view chrome is right. The project view needs to match Seed's bench-page layout: clear header, gradient blend at the sidebar boundary in the body. Export saves a backup in `.ledger/export/` so the user always has a recoverable copy.

   - In the body area (below the pill), the sidebar tan blends into the content cream via a horizontal gradient over ~20–30pt. The transition is softer here.

## (1) Item A — Project view chrome matches Seed bench pattern

### A1 — Layout structure

The project view, like Seed's bench, has:
- **Sidebar (left):** tan (`Brand.surfaceSidebar`), extends under title bar.
- **Right column:**
  - **Header band (top):** cream (`Brand.surfaceBase`), extends under title bar. Contains "Provenance" app label, the home/connect-project buttons. No tint, no separate header strip.
  - **Tab bar:** Overview / Sources / Artifacts / etc. — filter-chip styled per PR-24 — sits below the header band.
  - **Content area:** cream below the tab bar, with the project tiles + project context + manuscripts list.

The header band and content area are the same cream. There's no horizontal divider between them at the top of the right column — that's already in PR-24.

### A2 — Vertical seam: hard in header band, gradient in body

Where the sidebar's right edge meets the right column:

- **In the header band region (top ~40–48pt of the window, the height of the title bar + chrome row):** the sidebar tan ends crisply at its right edge. The right column cream starts crisply at that x. This is a **hard, clean vertical edge** — a dividing line.

- **In the body region (everything below the header band):** the sidebar tan blends into the content cream via a horizontal gradient. Implement as a thin gradient strip overlaid at the sidebar's right edge:

```swift
// In the right column's body container, at the leading edge:
LinearGradient(
    colors: [Brand.surfaceSidebar, Brand.surfaceBase],
    startPoint: .leading,
    endPoint: .trailing
)
.frame(width: 24)  // gradient width — adjust 20–30pt to match Seed
.frame(maxWidth: .infinity, alignment: .leading)  // anchor to leading edge of right column
```

The gradient sits ONLY in the body area, not in the header band. Visually: the user sees a crisp vertical line at the top (header zone) and a soft blend lower down (body zone).

### A3 — Implementation approach

A clean way to do this:

```swift
HSplitView {  // or NavigationSplitView equivalent
    // Sidebar
    SidebarView()
        .frame(minWidth: 220, idealWidth: 260)
        .background(Brand.surfaceSidebar.ignoresSafeArea())
    
    // Right column
    VStack(spacing: 0) {
        // Header band — top, NO gradient overlay
        HeaderBandView()
            .frame(height: 48)
            .background(Brand.surfaceBase.ignoresSafeArea(edges: .top))
        
        // Body — has the gradient seam at leading edge
        ZStack(alignment: .leading) {
            Brand.surfaceBase  // base
            LinearGradient(
                colors: [Brand.surfaceSidebar, Brand.surfaceBase],
                startPoint: .leading,
                endPoint: .trailing
            )
            .frame(width: 24)
            .frame(maxWidth: .infinity, alignment: .leading)
            
            VStack(spacing: 0) {
                TabBarView()
                ScrollView { ContentView() }
            }
        }
    }
}
```

The exact gradient width is a visual judgment call — start at 24pt, tune by eye against Seed's bench until the two apps look like siblings.

### A4 — Both modes

Test in light AND dark mode. `Brand.surfaceSidebar` and `Brand.surfaceBase` each have dark variants per PR-23 — the gradient should work in both modes because it's interpolating between two adaptive colors.

### A5 — Home view untouched

Don't change the Home view chrome. The single continuous background there stays as is. Only the project view changes.

## (2) Item B — Export dual-copy: backup to `.ledger/export/` + user-chosen location

### B1 — The new behavior

When the user exports (Review or Authorship Report, PDF or Markdown):

1. **Always write a backup copy to `.ledger/export/`** under the project's `.ledger/` folder. Filename includes timestamp and format, e.g. `2026-05-20T14-30-00_authorship.pdf` or `2026-05-20T14-30-00_review.md`.
2. **Then present an `NSSavePanel`** with a default filename like `{ProjectDisplayName}-review.pdf` and a default directory of the project root. The user picks where to save the "copy."
3. **Copy the bytes** from the `.ledger/export/` backup to the user's chosen location.

If the user cancels the save panel, the backup in `.ledger/export/` is still written — the user can always retrieve it later.

### B2 — The `.ledger/export/` folder

- Create on first export if it doesn't exist.
- Keep a history — don't overwrite. Each export adds a new timestamped file.
- The folder is included in `.ledger/` so it travels with the project.

### B3 — UI feedback

When export completes, show a brief confirmation (e.g., a toast or sheet) indicating both:
- "Saved to: {user's chosen path}"
- "Backup saved to: .ledger/export/{filename}"

If the user cancelled the save panel: just show "Backup saved to: .ledger/export/{filename}".

### B4 — Update Item F from PR-21

PR-21 introduced the NSSavePanel without a backup. PR-26 dropped the bundle. PR-27 amends the export flow to ALWAYS write the backup AND ask for the copy location. The save panel itself stays — only the dual-write behavior is new.

### B5 — Implementation sketch

```swift
func exportProject(intent: ExportIntent, format: ExportFormat) {
    // 1. Generate the file in-memory or to a temp location.
    let exportData = try generateExport(intent: intent, format: format)
    
    // 2. Write backup to .ledger/export/
    let backupURL = ledgerExportURL(for: intent, format: format, timestamp: .now)
    try FileManager.default.createDirectory(at: backupURL.deletingLastPathComponent(),
                                            withIntermediateDirectories: true)
    try exportData.write(to: backupURL)
    
    // 3. Present NSSavePanel for user copy
    let savePanel = NSSavePanel()
    savePanel.allowedContentTypes = [format.contentType]
    savePanel.directoryURL = projectURL
    savePanel.nameFieldStringValue = defaultExportFilename(intent: intent, format: format)
    
    savePanel.begin { response in
        if response == .OK, let userURL = savePanel.url {
            try? FileManager.default.copyItem(at: backupURL, to: userURL)
            showConfirmation(userPath: userURL, backupPath: backupURL)
        } else {
            // User cancelled — still show the backup confirmation
            showConfirmation(userPath: nil, backupPath: backupURL)
        }
    }
}

func ledgerExportURL(for intent: ExportIntent, format: ExportFormat, timestamp: Date) -> URL {
    let dateString = ISO8601DateFormatter().string(from: timestamp)
        .replacingOccurrences(of: ":", with: "-")
    let filename = "\(dateString)_\(intent.rawValue).\(format.extension)"
    return projectURL.appendingPathComponent(".ledger/export/\(filename)")
}
```
