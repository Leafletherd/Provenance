## (1) Item A — Sidebar selected row style

**The problem:** Provenance's sidebar selection looks different from Seed's. It may have a system-grey selection, drag reorder handles, or both.

**Target:** identical visual treatment to Seed's sidebar (SD-14 pattern):
- Selected row background: `prov/tint-surface` (#E4F2EE)
- 0.5pt left vertical stroke: `prov/tint-border` (#B0D8CC)
- Unselected rows: clear background
- No system list selection highlight (`.listRowBackground(.clear)` + `.listStyle(.sidebar)` or equivalent)
- **No drag handles / reorder controls** on sidebar items

### A1 — Remove drag handles

If the sidebar `List` uses `.onMove` or the rows have `.moveDisabled(false)`, remove them. The sidebar is not reorderable by the user — it reflects the project/artifact list as stored.

```swift
// Remove if present:
// .onMove { ... }
// ForEach(...).onMove { ... }
// .listRowReorderIndicator (or any handle overlay)
```

### A2 — Apply brand selection style

For each sidebar row:

```swift
.listRowBackground(
    isSelected
        ? Color("prov/tint-surface")
        : Color.clear
)
.overlay(alignment: .leading) {
    if isSelected {
        Rectangle()
            .fill(Color("prov/tint-border"))
            .frame(width: 0.5)
    }
}
```

Also suppress the system list selection:
```swift
// On the List:
.listStyle(.sidebar)
// On each row or the List:
.scrollContentBackground(.hidden)
```

If the sidebar uses `NSTableView` under the hood (via `NSViewRepresentable`), set:
```swift
tableView.selectionHighlightStyle = .none
```
and apply the background/stroke manually in the row view.

```
- [ ] A1. No drag handles visible on any sidebar item.
- [ ] A1. Selected row: prov/tint-surface background. No system grey.
- [ ] A1. 0.5pt prov/tint-border left stroke on selected row.
- [ ] A1. Unselected rows: clear background.
- [ ] A1. Consistent across project list and any other sidebar-style lists in Provenance.
```

## (2) Item B — Provenance-wide selection color audit

**The problem:** throughout Provenance, selected states may still use system blue or grey instead of `prov/accent` / `prov/tint-surface`.

Audit and fix every place a selected/active state is highlighted:

### B1 — List and table selection

Any `List`, `Table`, or `NSTableView` that allows row selection. Set selection highlight to use `prov/tint-surface` background + `prov/tint-border` stroke (same A2 pattern). No system blue row highlight.

### B2 — Segmented controls and pickers

If Provenance uses `Picker` style segmented control or custom tab-style selectors, the selected segment must use `prov/accent` as the active indicator color — not system blue or grey.

```swift
// Custom segment indicator:
Rectangle()
    .fill(Color("prov/accent"))
    .frame(height: 2) // underline style
```

Or if using `.pickerStyle(.segmented)`, override with:
```swift
.tint(Color("prov/accent"))
```

### B3 — Artifact detail / project header

If there is a selected artifact in a detail pane, its selection ring, highlight border, or background tint must be `prov/tint-surface` / `prov/tint-border`, not system blue.

### B4 — Focus rings

Any text field or area that shows a focus ring: set `.focusedBorderColor(Color("prov/tint-border"))` or suppress the system focus ring (`focusRingType = .none` on NSTextView/NSTextField) if the existing custom styling makes it redundant.

### B5 — Check list

Walk through the app:
- Home screen: project list selection ✓ (fixed in PR-19 — verify)
- Project view: artifact/item list selection
- Any filter/sort picker
- Any date or timeline picker selection state
- Tab bar or navigation selection indicator

For each: if it shows system blue or grey when selected, apply `prov/*` tokens.

```
- [ ] B1. No system blue or grey row selection highlight anywhere in Provenance.
- [ ] B2. Segmented controls / pickers use prov/accent for selected state.
- [ ] B3. Artifact detail selection uses prov/tint-surface or prov/tint-border.
- [ ] B5. Full walk-through confirmed: every selection state in the app uses prov/* tokens.
```

## (3) Item C — Home screen textarea cursor alignment

**The bug:** on the Provenance home screen, the text cursor in the text entry area appears at a different horizontal position than the placeholder text. The placeholder has visible left padding; the cursor is at the raw left edge.

This is the same bug that was fixed in Seed (SD-16, item 4). Apply the same fix.

**The fix:** ensure the NSTextView (or SwiftUI TextEditor) and its placeholder use identical left insets. Concretely:

If using NSTextView:
```swift
textView.textContainerInset = NSSize(width: 8, height: 8) // match to actual placeholder padding
textView.textContainer?.lineFragmentPadding = 0
```
And position the placeholder overlay at `x: 8, y: 8` (or equivalent padding values).

If using SwiftUI TextEditor with an overlay placeholder:
```swift
ZStack(alignment: .topLeading) {
    TextEditor(text: $text)
        .padding(.horizontal, 8)
    if text.isEmpty {
        Text(placeholderString)
            .padding(.horizontal, 12) // NSTextView internal padding is ~4pt; TextEditor + .padding(.horizontal, 8) = 12pt total
            .padding(.top, 8)
            .foregroundColor(Color(Brand.textMuted))
            .allowsHitTesting(false)
    }
}
```

**The correct padding value depends on the current implementation** — read it first, then match exactly. The goal is pixel-identical alignment.

```
- [ ] C1. Text cursor and placeholder text align to the same left position on the Provenance home screen.
- [ ] C1. Typed text starts at the same horizontal position as the placeholder.
- [ ] C1. Fix is consistent with how SD-16 fixed the same issue in Seed.
```
